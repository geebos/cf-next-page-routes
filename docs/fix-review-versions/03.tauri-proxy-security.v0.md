# 03 — Tauri Proxy Security & HTTP Adapter

**Priority:** High  
**Source:** [project-review.md](./project-review.md) findings #4, #7, #8; gaps G1, G2, G8  
**Category:** Security + Correctness (desktop networking)

---

## Problem

The Tauri stack has an open HTTP path and incomplete capability / adapter hardening:

| ID | Issue |
|----|--------|
| #4 | Rust `proxy` accepts any URL — intentional bypass of http plugin allowlist (SSRF) |
| G2 | `proxy` / `greet` not allowlisted in capabilities ACL — invoke may fail by default |
| G1 | `http:default` only allows `https://template.geebosblog.com/*` — local/other API hosts blocked for plugin-http |
| #7 | JS `request()` only nulls body for 204/304 — 205 throws in `Response` constructor |
| #8 | Multi `Set-Cookie` folded with `", "` — broken cookies |
| G8 | New `reqwest::Client` per call, no timeout — hung peers freeze UI |

---

## Goals

- No unrestricted SSRF from the webview.
- Capabilities match registered commands and intended hosts.
- Proxy response reconstruction matches Fetch null-body rules and safe header handling.
- Proxy requests time out instead of hanging forever.

## Implementation steps

### 1. Restrict `proxy` URLs (required)

In Rust, before `client.request`:

- Parse URL; reject non-http(s).
- Allow only configured prefixes (env / compile-time list / capability-aligned list), e.g.:
  - production API host
  - optional `http://127.0.0.1:*` / `http://localhost:*` for dev only (`cfg!(debug_assertions)` or feature flag)
- Reject link-local / metadata ranges if not explicitly needed (`169.254.0.0/16`, `metadata.google.internal`, etc.).

Document that Test page is not a general-purpose open proxy.

### 2. Capabilities ACL (required)

`src-tauri/capabilities/default.json`:

- Allow custom commands used by the frontend (`proxy`; drop or allow `greet` intentionally).
- Expand `http:default` `allow` URLs to every host `apiFetch` may hit:
  - production `NEXT_PUBLIC_BASE_URL` / `DEFAULT_TAURI_BASE_URL`
  - local worker if desktop dev uses plugin-http against localhost

Tauri 2: commands are denied unless listed — verify with `pnpm tauri dev` + `/test` and Todo list.

### 3. Null-body statuses in `request.ts` (required)

```ts
const NULL_BODY = new Set([101, 103, 204, 205, 304]);
return new Response(
  NULL_BODY.has(res.status) ? null : new Uint8Array(res.body),
  { status: res.status, headers: res.headers },
);
```

### 4. Set-Cookie / multi-value headers (required)

Options:

- **A.** Do not fold `set-cookie`; pass `Vec<(String, String)>` or separate `setCookies: Vec<String>` over IPC.
- **B.** Fold only non-sensitive headers; keep `set-cookie` as multiple map entries or array.

JS side: if multi-cookie, use `Headers.append` per cookie.

### 5. Timeouts + shared client (required + polish)

- Build one `Client` with `timeout` / `connect_timeout` (e.g. 30s / 10s).
- Reuse via `OnceLock` or `tauri::State`.

### 6. Optional efficiency

- Avoid `Array.from` entire body if a binary IPC path exists; not blocking for security batch.

## Acceptance criteria

- [ ] `invoke("proxy")` to `http://169.254.169.254/` (or other non-allowlisted host) returns error.
- [ ] Allowlisted production (and dev, if configured) hosts succeed from Test page.
- [ ] Capability config allows `proxy` when intended; `/test` works in Tauri when host is allowed.
- [ ] Todo `apiFetch` host is on `http:default` allowlist (or documented as sqlite-only desktop).
- [ ] Status 205 (and 204) reconstruct without throwing.
- [ ] Multi `Set-Cookie` not comma-joined into one invalid cookie string.
- [ ] Request to non-responding host fails within timeout, not infinite spinner.
