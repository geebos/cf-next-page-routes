# 02 — Due Date Timezone Consistency

**Priority:** High  
**Source:** [project-review.md](./project-review.md) findings #2, #3, #14  
**Category:** Correctness (domain dates)

---

## Problem

Todo `dueDate` is stored as **UTC midnight of a calendar day**, but create UI and list display mix **local** and **UTC** clocks:

| Surface | Current behavior | Problem |
|---------|------------------|---------|
| Form `onSelect` | `Date.UTC(localY, localM, localD)` | OK as “local calendar day as UTC midnight” |
| Form `disabled` | `{ before: new Date() }` | Local “now”, not same day boundary as schema |
| `createTodoSchema` refine | `utcStartOfTodayMs()` (UTC day) | Rejects local “today” in Americas evening |
| List `format` | `format(new Date(ms), "yyyy-MM-dd")` local | West of UTC shows previous day |
| List overdue | `dueDate < utcStartOfTodayMs()` | UTC; can disagree with displayed label |
| `updateTodoSchema` | any `int` dueDate | Allows deliberate backdate (`dueDate: 0`) |

---

## Goals

- One shared calendar-day model for encode / disable / validate / display / overdue.
- Create cannot pick a past **calendar day** under the chosen model.
- Update policy is explicit: either allow keeping existing past, or reject newly chosen past dates.

---

## Recommended model

**Local calendar day as product truth** (typical for personal todos):

1. **Store:** still `Date.UTC(y, m, d)` where `y/m/d` are the user’s local calendar day of the picked date (current form encode — keep).
2. **“Today” for validation / disabled:** local start-of-today as UTC midnight of *local* YMD:

```ts
export function localStartOfTodayMs(now = new Date()): number {
  return Date.UTC(now.getFullYear(), now.getMonth(), now.getDate());
}
```

3. **Display:** format using **UTC** parts of the stored ms (because value is UTC midnight of that calendar day):

```ts
export function formatDueDateMs(ms: number): string {
  const d = new Date(ms);
  const y = d.getUTCFullYear();
  const m = String(d.getUTCMonth() + 1).padStart(2, "0");
  const day = String(d.getUTCDate()).padStart(2, "0");
  return `${y}-${m}-${day}`;
}
```

4. **Overdue:** `!completed && dueDate < localStartOfTodayMs()`.

Alternatively, pure-UTC product calendar (worker TZ-agnostic) is fine if form disabled + default + display all use UTC helpers consistently — pick one model and apply everywhere.

## Implementation steps

### 1. Shared helpers in schema module

In `src/shared/schemas/todo.ts` (or small `src/shared/date.ts` re-exported):

- `localStartOfTodayMs()` (or `utcStartOfTodayMs` if pure-UTC model)
- `calendarDayToMs(y, m, d)` → `Date.UTC(y, m, d)`
- `formatDueDateMs(ms)` → `yyyy-MM-dd` from UTC parts
- Keep or replace `utcStartOfTodayMs` so call sites do not diverge

### 2. Align create schema refine

```ts
dueDate: z
  .number()
  .int()
  .refine((n) => n >= localStartOfTodayMs(), "不能选过去日期"),
```

Ensure form default uses the same helper as refine.

### 3. Align form Calendar

- `onSelect`: keep mapping local YMD → `Date.UTC` (or shared helper).
- `disabled`: `{ before: new Date(localStartOfTodayMs()) }` or matcher using same “today” as schema (not raw `new Date()` wall clock alone if that mismatches day boundary).
- Display button label: `formatDueDateMs(field.value)`.
- `selected`: Date that day-picker treats as that calendar day under the chosen model (document carefully for react-day-picker).

### 4. Align list

- Label: `formatDueDateMs(todo.dueDate)`.
- Overdue: same “today” helper as create.

### 5. Update past-date policy

Choose product rule:

| Rule | Implementation |
|------|----------------|
| **Keep past on edit only if unchanged** | On PATCH, if `input.dueDate` is set and `< today`, reject unless equal to current row’s dueDate |
| **Never past on create or update** | Apply same refine on update when `dueDate` is present |
| **Allow any past on update** | Keep schema; document only (not recommended given create rule) |

Recommended default: **same min-day refine on update when dueDate is provided**, so `PATCH { dueDate: 0 }` fails, while omitting dueDate leaves existing past value.

## Acceptance criteria

- [ ] In a west-of-UTC TZ near local evening after UTC midnight: picking local “today” validates on client and server.
- [ ] Stored `Date.UTC(2026, 6, 20)` displays as `2026-07-20` in America/Los_Angeles (no off-by-one).
- [ ] Overdue badge matches the displayed calendar day under the same “today” definition.
- [ ] `PATCH` with a newly chosen past `dueDate` is rejected (if policy chosen above).
- [ ] Existing past dueDate can still be loaded in edit form and saved without changing the date (if keep-unchanged policy).

## Out of scope

- Server timezone config for CF Workers (use pure functions of `Date` only).
- Non-todo date fields.
