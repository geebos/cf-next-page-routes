### R02-001 — 推荐的 local「今天」在 Worker 上等于 UTC，服务端校验与客户端会分叉

- **问题编号:** R02-001
- **提出版本:** v0
- **问题描述:** 文档推荐 `localStartOfTodayMs()` 用 `now.getFullYear/Month/Date()`（运行时本地日历日）。相关代码里 create 校验跑在共享 schema 上：客户端（用户浏览器 TZ）与 CF Worker（通常为 UTC）都会调用同一函数。Worker 侧「local」≈ UTC，因此美洲晚间用户在客户端选的 local today 可能通过表单，却在 `POST /api/todos` 被 `createTodoSchema` 以 UTC today 拒绝（或反向：UTC 已跨日、本地尚未跨日时两边不一致）。现有 schema 注释明确写了「compares … regardless of the worker's local timezone」，与推荐模型直接冲突。文档只在 Out of scope 写「Server timezone config… use pure functions of Date only」，未解决「同一 helper 在 client/server 语义不同」的核心问题。
- **Review Agent 建议:** 必须先定产品真值落在哪一侧，并写清服务端如何拿到「用户日历日」：(a) 纯 UTC 产品日（保持 `utcStartOfTodayMs`，表单 encode/display/disabled 全跟 UTC）；(b) 用户 local 产品日，但服务端不复算 local today，改为校验「ms 必须是 `Date.UTC(y,m,d)` 形态 + 可选 trust client / 传 IANA TZ / 只做粗范围」；(c) 仅客户端挡过去日、服务端只校验类型与合理上下界。任选其一写进 Recommended model，并删掉「同一 `localStartOfTodayMs` 到处用」的隐含假设。
- **Answer Agent 回复:** 采纳 **(b)**：产品真值 = 用户本地日历日；存储仍为该日 YMD 的 `Date.UTC(y,m,d)`；展示一律用 UTC 部件。`localStartOfTodayMs` **仅允许在浏览器端**（form default / disabled / list overdue）。共享 schema 与 Worker **禁止**用 `getFullYear/Month/Date` 求「今天」。服务端 create/update 的 past 相关校验只做两件事：(1) `isCalendarMidnightMs(n)`（`n === Date.UTC(UTC-y,m,d)`）；(2) 可选宽松下界 `n >= Date.now() - LOOSE_PAST_MS`（默认 48h），用于挡 `dueDate: 0` 等垃圾值，**不是**用户日历「今天」。客户端 refine 可继续用 `localStartOfTodayMs` 做 UX；若共享 schema 必须 client/server 共用，则 schema 只含 midnight + loose bound，local 日界校验放在 form resolver 或客户端专用 refine。已删除「同一 `localStartOfTodayMs` 到处用」假设。
- **文档调整说明:** Recommended model 重写为 Client/Server 分责表；去掉 Worker 上调用 local today 的表述；Goals / Implementation / Out of scope 对齐。
- **当前状态:** 已修改
- **关闭版本:** v1

### R02-003 — Update past-date 策略自相矛盾，且与 edit 表单「整包提交」不兼容

- **问题编号:** R02-003
- **提出版本:** v0
- **问题描述:** §5 推荐默认「update 在 dueDate 出现时套同一 min-day refine」，同时又列「Keep past on edit only if unchanged」。Acceptance 同时要求：(1) 新选 past 的 PATCH 拒绝；(2) 已有 past dueDate 在 edit 中不改日期仍可保存。现码 `UpdateTodoForm` 始终提交完整 `FormValues`（含 `dueDate`），`updateTodoSchema` 注释写的是 “edit allows keeping an existing past due date (no past-date check here)”。若仅在 schema 上对 `dueDate` 做 `>= today`，则「打开过期任务点保存、日期未改」会 400，与 AC (2) 冲突；若要 keep-unchanged，schema refine 看不到「当前行」，必须在 `worker/routes/todos.ts` PATCH 里读 row 再比较——但 Files to touch 未列 worker 路由。
- **Review Agent 建议:** 只保留一条产品规则并写清实现落点：若选 keep-unchanged → 必须改 worker PATCH（读现有 `dueDate`，仅当 `input.dueDate !== existing && input.dueDate < today` 拒绝），schema 侧不能单独 `>= today`；若选 never-past → 同步改 AC，明确「过期任务必须先改到今天或以后才能保存」，并说明 edit 表单是否禁用过去日。删除「推荐 A 但 AC 写 B」的双轨表述。
- **Answer Agent 回复:** **只保留 keep-unchanged**。`updateTodoSchema` 不设 local-today min-day；仅做 int +（若有 dueDate）`isCalendarMidnightMs` + loose bound。真正「新选 past 拒绝 / 未改 past 放行」在 **worker PATCH**：先读 row；若 `input.dueDate !== undefined && input.dueDate !== existing.dueDate`，则对 new value 做与 create 相同的 **客户端产品规则无法在 server 完美复现时**：server 用 midnight + loose bound；**客户端** edit 在用户**改日**时用 `localStartOfTodayMs` 挡过去日（Calendar `disabled` 在 edit 也启用，但当前值若已 past 仍保持 selected 可提交）。为匹配 AC「新选 past 的 PATCH 拒绝」：worker 在 dueDate **变更**时，额外要求 `input.dueDate >= Date.now() - LOOSE_PAST_MS` 且 midnight——这挡 `0` 与明显过期垃圾，但不伪装成用户日历 today。**产品层「不能新选昨天」以客户端为准**；server 防篡改用 loose bound。已删 never-past 双轨与「update 套同一 min-day refine」推荐。
- **文档调整说明:** §5 只保留 keep-unchanged；Files/Steps 标明 `todos.ts` PATCH 必改；AC 去掉 never-past 歧义。

## Problem

Todo `dueDate` is stored as **UTC midnight of a calendar day**, but create UI and list display mix **local** and **UTC** clocks:

| Surface | Current behavior | Problem |
|---------|------------------|---------|
| Form `onSelect` | `Date.UTC(localY, localM, localD)` | OK as “local calendar day as UTC midnight” |
| Form `disabled` | `{ before: new Date() }` | Local “now”, not same day boundary as schema |
| Form `selected` | `new Date(field.value)` | UTC midnight interpreted as local wall → off-by-one west of UTC |
| Form button / list label | `format(new Date(ms), "yyyy-MM-dd")` | Local parts → west of UTC shows previous day |
| `createTodoSchema` refine | `utcStartOfTodayMs()` (UTC day) | Rejects local “today” in Americas evening; also wrong if reused as “local” on Worker |
| List overdue | `dueDate < utcStartOfTodayMs()` | UTC day; can disagree with displayed label |
| `updateTodoSchema` | any `int` dueDate | Allows deliberate backdate (`dueDate: 0`); edit form always resubmits full `dueDate` |

---

## Goals

- One product calendar model: **user local calendar day**, stored as UTC midnight of that YMD.
- Encode / picker selected / disable / client validate / display / overdue all use the same helpers.
- Server never pretends to know the user’s TZ: only **calendar-midnight shape** + **loose absolute bound**.
- Create cannot pick a past **local** calendar day (client-enforced).
- Update policy is **keep-unchanged**: resubmitting an existing past dueDate succeeds; **changing** to a non-midnight or absurd past value fails server-side; client blocks newly picking a past local day on edit too.

---

## Recommended model (locked)

**Product truth = user local calendar day.**  
**Wire format = ms at `Date.UTC(y, m, d)` for that day.**  
**Display = UTC Y/M/D of the stored ms** (so the stored “day number” is what users see).

| Layer | “Today” / past rule | Allowed helpers |
|-------|---------------------|-----------------|
| **Browser only** | Local calendar today | `localStartOfTodayMs(now?)`, `localStartOfTodayDate(now?)` |
| **Shared schema + Worker** | No local today | `isCalendarMidnightMs(n)`, optional `n >= Date.now() - LOOSE_PAST_MS` |
| **Display (any runtime)** | N/A | `formatDueDateMs(ms)` via **UTC** parts only |
| **Picker bridge (browser)** | N/A | `msToPickerDate` / `pickerDateToMs` |

### Store

Keep current form encode:

```ts
export function calendarDayToMs(y: number, m: number, d: number): number {
  return Date.UTC(y, m, d);
}

export function pickerDateToMs(d: Date): number {
  return calendarDayToMs(d.getFullYear(), d.getMonth(), d.getDate());
}
```

### Client “today”

```ts
/** Browser only. Do not call on Worker (Worker "local" ≈ UTC). */
export function localStartOfTodayMs(now = new Date()): number {
  return Date.UTC(now.getFullYear(), now.getMonth(), now.getDate());
}

/** Local wall-clock midnight Date for day-picker `disabled.before`. */
export function localStartOfTodayDate(now = new Date()): Date {
  return new Date(now.getFullYear(), now.getMonth(), now.getDate());
}
```

### Server / shared shape checks

```ts
export const LOOSE_PAST_MS = 48 * 60 * 60 * 1000;

export function isCalendarMidnightMs(n: number): boolean {
  if (!Number.isInteger(n)) return false;
  const d = new Date(n);
  return n === Date.UTC(d.getUTCFullYear(), d.getUTCMonth(), d.getUTCDate());
}

export function isDueDateInLooseRange(n: number, now = Date.now()): boolean {
  // Upper bound optional: e.g. now + 10y — omit if not needed for v1
  return n >= now - LOOSE_PAST_MS;
}
```

- **Create schema (shared, runs on Worker + client resolver):**  
  `dueDate` int + `isCalendarMidnightMs` + `isDueDateInLooseRange` (message can stay generic / 不能选过去日期 for loose failures that look like past).
- **Create form (browser):** additionally ensure default = `localStartOfTodayMs()`, Calendar disabled with `localStartOfTodayDate()`, and prefer client-side guard `dueDate >= localStartOfTodayMs()` so users cannot submit local yesterday even if still inside 48h loose window.

### Display

```ts
export function formatDueDateMs(ms: number): string {
  const d = new Date(ms);
  const y = d.getUTCFullYear();
  const m = String(d.getUTCMonth() + 1).padStart(2, "0");
  const day = String(d.getUTCDate()).padStart(2, "0");
  return `${y}-${m}-${day}`;
}
```

### Picker selected / disabled (mandatory)

```ts
/** UTC YMD of stored ms → local Date at local midnight (picker selection day). */
export function msToPickerDate(ms: number): Date {
  const d = new Date(ms);
  return new Date(d.getUTCFullYear(), d.getUTCMonth(), d.getUTCDate());
}
```

| Prop | Value |
|------|--------|
| `selected` | `field.value ? msToPickerDate(field.value) : undefined` |
| `onSelect` | `(d) => d && field.onChange(pickerDateToMs(d))` |
| `disabled` | `{ before: localStartOfTodayDate() }` for **create and edit** |
| Button / list label | `formatDueDateMs(ms)` — **never** `format(new Date(ms), "yyyy-MM-dd")` |

**Why not `disabled: { before: new Date(localStartOfTodayMs()) }`?**  
`localStartOfTodayMs()` is UTC midnight of local YMD. In America/Los_Angeles that instant is still **local previous evening**, so day-picker may treat “today” as before the boundary and disable today.

### Overdue (browser list)

```ts
const overdue = !todo.completed && todo.dueDate < localStartOfTodayMs();
```

### Update policy: keep-unchanged (only rule)

| Case | Client | Server |
|------|--------|--------|
| Edit, dueDate unchanged (including existing past) | Allow submit | PATCH: if `input.dueDate === existing.dueDate`, accept (no loose/past re-check beyond type) |
| Edit, dueDate changed | Calendar disables local past days; resolver can require `>= localStartOfTodayMs()` | PATCH: if changed, require `isCalendarMidnightMs` + `isDueDateInLooseRange`; reject with 400 |
| `PATCH { dueDate: 0 }` | N/A | 400 (not midnight and/or outside loose range) |
| Omit `dueDate` on PATCH | N/A | leave existing value |

`updateTodoSchema`: optional int; if present, midnight + loose range (so API clients cannot set `0`). Worker PATCH still must **load the row** to allow equal past values that fail loose range (e.g. dueDate older than 48h kept on edit).

Worker sketch:

```ts
// after validate body, before apply patch
if (input.dueDate !== undefined) {
  const existing = await db.select()...get();
  if (!existing) throw new BusinessError("任务不存在", 404);
  if (input.dueDate !== existing.dueDate) {
    if (!isCalendarMidnightMs(input.dueDate) || !isDueDateInLooseRange(input.dueDate)) {
      throw new BusinessError("不能选过去日期", 400);
    }
  }
}
```

If schema already enforces midnight + loose on every provided `dueDate`, equal-past values older than 48h would fail schema **before** handler — **therefore** `updateTodoSchema` must **not** apply loose/past on the field alone; only int (+ optional midnight for *shape* if we always want midnight storage). Prefer:

- `updateTodoSchema.dueDate`: `z.number().int().optional()` + refine midnight only (always store calendar midnights).
- Worker PATCH: if changed → also loose range; if unchanged → accept even if outside loose range.

### Day-boundary / clock skew

1. Helpers take optional `now` for tests.
2. Client validates at **submit time** with `localStartOfTodayMs()`; stale default after local midnight → refine error → user re-picks.
3. Server does **not** recompute user-local today → no client/Worker TZ fork for “is today valid”.
4. No `today - 1 day` grace window (would legalize yesterday).
5. UX: keep message `不能选过去日期`; no special cross-midnight toast required for v1.

## Implementation steps

### 1. Shared helpers in `todo.ts`

Add symbols from Recommended model. Delete `utcStartOfTodayMs`. Document browser-only on local today helpers.

### 2. Create schema

```ts
dueDate: z
  .number()
  .int()
  .refine(isCalendarMidnightMs, "不能选过去日期")
  .refine((n) => isDueDateInLooseRange(n), "不能选过去日期"),
```

Client form still uses `localStartOfTodayMs` for default and Calendar disable so local yesterday cannot be chosen in UI even when inside 48h.

### 3. Form Calendar

- `selected={field.value ? msToPickerDate(field.value) : undefined}`
- `onSelect` → `pickerDateToMs`
- `disabled={{ before: localStartOfTodayDate() }}` for create **and** edit
- Label: `formatDueDateMs(field.value)`
- Create default: `dueDate: localStartOfTodayMs()`

### 4. List

- Label: `formatDueDateMs(todo.dueDate)`
- Overdue: `todo.dueDate < localStartOfTodayMs()`

### 5. Worker PATCH keep-unchanged

- Load row; 404 if missing.
- If `input.dueDate !== undefined && input.dueDate !== existing.dueDate`: require midnight + loose; else 400 `不能选过去日期`.
- If equal: accept (past-beyond-loose kept).
- Confirm POST still goes through `jsonValidator(createTodoSchema)` → 400 on bad dueDate.

### 6. Verify dual path

- POST past / non-midnight → 400.
- PATCH `dueDate: 0` → 400.
- PATCH same past dueDate on overdue todo → 200.
- Grep clean for old helpers / local `format` on dueDate.
