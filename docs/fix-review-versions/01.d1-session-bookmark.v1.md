### R01-001
- **问题编号:** R01-001
- **提出版本:** v0
- **问题描述:** Write 路径只写了 `if (next) setCookie else omit`，未覆盖浏览器里**已经存在**的 `d1_bm=` 空串 cookie。omit 不会清掉旧空值；下一请求仍会读到 `""` 并走 `withSession("")`，除非 read 侧严格 coalesce。文档把 read coalesce 和 write omit 写成二选一风格，未明确“双端都必须做 + 是否 deleteCookie 迁移脏 cookie”。
- **Review Agent 建议:** 强制要求：(1) read 用 `raw || DEFAULT_BOOKMARK`（或等价空串/缺失统一回退）；(2) write 仅在 `next` 为非空时 `setCookie`；(3) 若 `getBookmark()` 为 null/空且请求 cookie 为 `""`，调用 `deleteCookie` 清脏状态。在实现步骤写死，不要用注释式“Prefer”留歧义。
- **Answer Agent 回复:** 采纳。三端强制且缺一不可：(1) read：`const inbound = getCookie(...) ?? ""`，`const bookmark = inbound || DEFAULT_BOOKMARK`；(2) write：仅当 `next` 非空时 `setCookie`；(3) 迁移：若 `!next && inbound === ""`，`deleteCookie` 清脏。omit  alone 不够。见 Implementation step 1。
- **文档调整说明:** Step 1 重写为强制三分支伪代码；Problem #1 补充脏 cookie 迁移；Acceptance 增加 deleteCookie 场景。

## Problem

D1 read-your-writes depends on the `d1_bm` bookmark cookie. Three related defects break that path:

1. **Empty string overwrite + dirty cookie** — `session.getBookmark() ?? ""` writes `""` when bookmark is null. Next request uses `getCookie(...) ?? DEFAULT`, but `""` is not nullish, so `withSession("")` runs instead of `"first-unconstrained"`. Omitting Set-Cookie alone does **not** clear a browser cookie that is already `d1_bm=` (empty); read-side coalesce **and** write-side delete of the dirty empty value are both required.
2. **Always `secure: true`** — browsers refuse Secure cookies on `http://localhost`, so local dev never persists the cookie through the Next rewrite path.
3. **Tauri remote default + health wipe chain** — Shipped Tauri defaults to remote D1 (`DEFAULT_TAURI_BASE_URL = "https://template.geebosblog.com"`, capabilities allow only that host). On launch `_app.tsx` calls `warmupNetworkPermission()` → `GET /api/health`. Health is mounted under `app.use("*", createDbMiddleware())` and does not run SQL, yet still runs `withSession` + unconditional `setCookie(... ?? "")`. Sequence **create → warmup health → list** can erase a good bookmark with `""`. Separately, Tauri WebView origin ≠ remote API origin; with `SameSite=Lax` the bookmark cookie is not a reliable RYW channel for desktop remote (see Step 3 / decision **C**).

---

## Goals

- Empty or missing inbound cookie always yields `withSession("first-unconstrained")`.
- Never write an empty-string `d1_bm` cookie; delete an existing empty dirty cookie when present.
- `secure` is true **only** when `new URL(c.req.url).protocol === "https:"` (no `x-forwarded-proto`, no `NODE_ENV`).
- **Tauri decision (locked):** default **C** — remote desktop does **not** guarantee RYW via `d1_bm`. Recommended local desktop path is **A** (sqlite worker). Option **B** (cross-origin cookie jar) is out of scope for this change.
- Document the above in README so implementers do not assume desktop RYW against remote D1.

## Implementation steps

### 1. Coalesce empty bookmark + write policy (required)

**Read** (always):

```ts
import { getCookie, setCookie, deleteCookie } from "hono/cookie";

const inbound = getCookie(c, BOOKMARK_COOKIE) ?? "";
const bookmark = inbound || DEFAULT_BOOKMARK; // "" and missing both → first-unconstrained
const session = c.env.DB.withSession(bookmark);
// ...
await next();
```

**Write** (after `next()`, d1 branch only):

```ts
const next = session.getBookmark(); // string | null | undefined — confirm with runtime
if (next && next !== inbound) {
  setCookie(c, BOOKMARK_COOKIE, next, cookieOptions(c));
} else if (!next && inbound === "") {
  // Migrate dirty empty cookie left by older builds
  deleteCookie(c, BOOKMARK_COOKIE, { path: "/" });
}
// else: omit Set-Cookie entirely
//   - next empty but inbound was a real bookmark → do not wipe
//   - next === inbound → no need to re-set
```

**Hard rules (not preferences):**

1. Read **must** treat missing and `""` as `DEFAULT_BOOKMARK`.
2. Write **must not** use `?? ""` as cookie value.
3. Write **must** `setCookie` only when `next` is non-empty **and** differs from `inbound`.
4. Write **must** `deleteCookie` when inbound was the dirty empty string and `next` is empty/null.
5. Before coding, **measure once** on local wrangler: hit `GET /api/health` with no prior cookie and log `session.getBookmark()` after `next()`. Expect null/empty when no statements ran; if a non-empty token appears, the `next !== inbound` guard still prevents pointless churn.

### 2. Conditional `secure` (required)

```ts
function cookieOptions(c: Context) {
  const isHttps = new URL(c.req.url).protocol === "https:";
  return {
    httpOnly: true,
    secure: isHttps, // ONLY source — do not read x-forwarded-proto or NODE_ENV
    sameSite: "lax" as const,
    path: "/",
    maxAge: 60 * 60,
  };
}
```

- Local `http://localhost:8787` / rewrite via `http://localhost:3000` → `secure: false` → browser stores cookie.
- Production HTTPS URL → `secure: true`.
- Forged or missing `x-forwarded-proto` must **not** flip `secure` (header is ignored).

## Acceptance criteria

- [ ] After `GET /api/health` with a previously valid `Cookie: d1_bm=<token>`, response does **not** set `d1_bm=` (empty) and does **not** clear a non-empty token (no destructive Set-Cookie / no delete of valid value).
- [ ] Inbound cookie missing **or** `d1_bm=` (empty) → `withSession("first-unconstrained")`.
- [ ] Inbound dirty empty cookie + null/empty `getBookmark()` → response deletes `d1_bm` (migration).
- [ ] Browser path under `pnpm dev`: after a write API via `http://localhost:3000/api/...`, DevTools shows `d1_bm` on host **`localhost:3000`**, and a subsequent same-origin list request resends it.
- [ ] Direct `http://localhost:8787` can also store a non-Secure `d1_bm` (middleware unit of work); this is debug-only, not the frontend path.
- [ ] When `c.req.url` is `https:`, Set-Cookie includes `Secure`; forged/missing `x-forwarded-proto` does not force `Secure=false` on https URL.
- [ ] When `c.req.url` is `http:`, Set-Cookie does **not** include `Secure`.
- [ ] Tauri decision **C** written in this doc + README; no requirement to verify remote desktop RYW. Optional note: A (`dev:sqlite`) for desktop RYW.
