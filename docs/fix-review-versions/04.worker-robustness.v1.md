## Problem

| ID | Issue |
|----|--------|
| #5 | `createErrorHandler` returns raw `err.message` on 500 → config/SQL leak to clients |
| #11 | `getSqliteDb` TOCTOU: concurrent first requests double-open + double-migrate |

## Goals

- Clients never see internal exception text on 500; public body is stable (`code: 500` / INTERNAL_ERROR semantics + generic message).
- SQLite engine initializes exactly once under concurrency; failed init is retryable.
- Node entry is hard to misconfigure (fail-fast); migrations resolve via env or cwd.
- Concurrent sqlite writes wait (`busy_timeout`) instead of hard-failing when possible; WAL best-effort on local disk.

## Implementation steps

### 1. Generic 500 responses (required)

Public body is **stable** and non-leaking. Wire shape stays compatible with `BusinessError` / `parseError` (`code` is number):

```ts
// InternalError branch
console.error("[InternalError]", {
  kind: "InternalError",
  method: c.req.method,
  path: c.req.path,
  msg: err instanceof Error ? err.message : String(err),
}, err);
return c.json(
  { code: 500, message: "服务器错误", data: null },
  500,
);
```

- Semantic name for this path: **INTERNAL_ERROR** (`code === 500` internal, not business).
- Chinese generic `message` is OK for now; frontend may later map by `code` (out of scope for this doc).
- Keep full detail only in logs. Do **not** return `err.message` / stack / paths / SQL to clients.
- `BusinessError` path: same structured log fields (`kind: "BusinessError"`, `method`, `path`, plus `code`), response body unchanged (`{ code: err.code, message: err.message, data: null }`).

**Required log fields (both branches):** `kind`, `method`, `path`, `msg`. Prefer object form over unescaped string interpolation. Optional: `engine` via `getEngine()`.

### 2. SQLite init singleton (required)

```ts
let sqliteDb: BetterSQLite3Database<typeof schema> | null = null;
let sqliteInit: Promise<BetterSQLite3Database<typeof schema>> | null = null;

export async function getSqliteDb() {
  if (sqliteDb) return sqliteDb;
  if (!sqliteInit) {
    sqliteInit = (async () => {
      let conn: import("better-sqlite3").Database | undefined;
      try {
        // resolve dir, mkdir, open conn, drizzle(conn), pragmas, migrate
        // ...
        sqliteDb = db; // ONLY after open + migrate success
        return db;
      } catch (e) {
        try {
          conn?.close();
        } catch {
          // ignore close errors
        }
        sqliteInit = null; // required: allow later retry
        throw e;
      }
    })();
  }
  return sqliteInit;
}
```

**Invariants (required):**

- `sqliteDb !== null` ⇒ connection open **and** migrate succeeded.
- All concurrent first callers share the same in-flight `sqliteInit` promise.
- Assign `sqliteDb` only on the success path (after migrate).
- On failure: `sqliteDb` stays `null`, `sqliteInit` is cleared (**required**, not optional), half-open `conn` is closed.

### 3. `serve.ts` fail-fast (required)

**Do not** silently default `DB_ENGINE` to `sqlite` inside serve. `getEngine()` remains the single source of truth (unset → `"d1"`).

Startup sequence before `listen`:

1. `const engine = getEngine()`.
2. If `engine === "d1"`: log actionable error — Node serve does not support D1; use wrangler **or** set `DB_ENGINE=sqlite` (+ `DB_SQLITE_DIR`) — then **exit / refuse listen**.
3. If `engine === "sqlite"` and `!process.env.DB_SQLITE_DIR`: log missing `DB_SQLITE_DIR` — **refuse listen**.
4. If `engine === "sqlite"` and dir set: optionally warm `await getSqliteDb()` so migrate failures surface at boot; then listen.
5. Startup log must print `engine=${getEngine()}` (or the same value), never a serve-local fake default.

### 4. Migrations path (required)

Resolve only inside the Node / `getSqliteDb` dynamic-import path (must not ship into Workers bundle):

1. If `process.env.DB_MIGRATIONS_DIR` is set → use it (absolute, or relative to `process.cwd()`).
2. Else → `path.join(process.cwd(), "drizzle")`.

**Do not** use `import.meta.url` / `fileURLToPath` as the default path (fragile under bundlers). Document that operators either start from repo root or set `DB_MIGRATIONS_DIR`.

### 5. SQLite pragmas (recommended, with fallback)

After open, before migrate:

```ts
conn.pragma("busy_timeout = 5000"); // always
try {
  conn.pragma("journal_mode = WAL"); // recommended on local disk
} catch (e) {
  console.warn("[sqlite] WAL unavailable, keeping default journal_mode", e);
}
```

- `busy_timeout` is always set.
- WAL is best-effort: network FS / some Docker volumes / sync drives may fail — warn and continue.
- Multi-process writers can still hit busy; single-process Node serve busy usually means concurrent external tools.

## Acceptance criteria

- [ ] Forced internal throw → client body is `{ code: 500, message: "服务器错误", data: null }` (or same stable fields); never stack/config/SQL/path text.
- [ ] 500 / BusinessError logs include at least `kind`, `method`, `path`; response still has no internal detail.
- [ ] Parallel first requests under sqlite: single open+migrate; stable `sqliteDb`.
- [ ] Init failure clears `sqliteInit`; next request retries open (e.g. bad dir then fixed, or inject).
- [ ] Node serve with unset/`d1` engine refuses listen with actionable log (not opaque per-request 500).
- [ ] Node serve with `sqlite` but missing `DB_SQLITE_DIR` refuses listen.
- [ ] Migrations apply with default `process.cwd()/drizzle` from repo root, **or** with `DB_MIGRATIONS_DIR` when cwd ≠ repo root.
- [ ] Rapid concurrent toggles under sqlite rarely/no longer return raw `SQLITE_BUSY` when timeout wait suffices; WAL failure does not block startup.

## Out of scope

- D1 bookmark cookies ([01](./01.d1-session-bookmark.md)).
- Zod validation message localization.
- Server-side localization of 500 / INTERNAL_ERROR via `Accept-Language` (frontend maps by `code` later).
- Sharing one sqlite file across hosts / network FS as a supported deployment.

## Test plan

1. Temporarily throw in a route → JSON body is generic 500; log line includes `path` (and `method`/`kind`); body has no internal detail.
2. `ab` / parallel `fetch` against cold sqlite worker → one migrate, healthy responses.
3. Start serve without env / with `DB_ENGINE=d1` → clear refusal before listen.
4. Start with `DB_ENGINE=sqlite` but no `DB_SQLITE_DIR` → clear refusal.
5. Start with wrong cwd and no env → migrate fails or misses folder per docs; set `DB_MIGRATIONS_DIR` absolute → migrate succeeds.
6. Force first init failure (bad path), then fix env and retry a request → second attempt re-opens (proves `sqliteInit` clear).
