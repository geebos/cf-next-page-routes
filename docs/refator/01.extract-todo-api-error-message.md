# 01.extract-todo-api-error-message

## 发现的问题

`src/pages/[locale]/todo.tsx` 中 5 处 catch 使用相同的错误消息选择逻辑：

```ts
e instanceof ApiError && e.message ? e.message : fallback
```

分别用于 `refresh` / `handleCreate` / `handleToggle` / `handleUpdate` / `handleDelete`，fallback 为不同 i18n key。重复增加维护成本，且使 `TodoPage` 圈复杂度偏高（pi-lens 报告 23）。

## 问题证据和影响范围

| 项 | 说明 |
| ---- | ------ |
| 证据 | 5 处字面重复；pi-lens high-complexity on `TodoPage` |
| 影响文件 | 仅 `src/pages/[locale]/todo.tsx` |
| 外部行为 | 不变：空 `message` 仍走 fallback；`handleDelete` 仍 rethrow |

## 原始设计

各 handler 内联三元表达式，无共享 helper。

## 最小修改方案

同文件模块顶层增加私有函数：

```ts
function apiErrorMessage(e: unknown, fallback: string): string {
  return e instanceof ApiError && e.message ? e.message : fallback;
}
```

5 处 catch 改为调用该函数。不抽 hook/lib，不改 List/Form/Dialog，不改数据流。

## Design Challenger 结论

- **verdict:** `APPROVE`
- **risks:** 空 `message` 必须保留 `&& e.message`；避免过度测试
- **simpler-alternative:** 不做更简但会保留重复；同文件 helper 为下限
- **required-tests:** typecheck + lint；可选 3 条纯函数断言（本轮未加）

备注：此前 `api.ts` 双 helper / `assertOk` 方案均被 **REJECT**（无真实问题），改选本项。
