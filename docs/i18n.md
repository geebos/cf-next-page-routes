# i18n 实现指南

适用于本项目当前栈：Next.js 16 Pages Router + `output: 'export'` + `react-i18next`（合并格式文件 + `pickLocale` 加载转换，不依赖 `next-i18next`） + 手动 `src/pages/[locale]` 路由 + TypeScript + pnpm + Cloudflare Worker + Tauri。

## 静态导出的约束

`output: 'export'` **不支持**：

- Next.js 内置 Internationalized Routing（即 `next.config.ts` 的 `i18n` 字段）
- Redirect / Rewrite / Proxy / Middleware（`rewrites()` 仅 `next dev` 阶段生效，`next build` 产物是纯静态文件）
- `getServerSideProps`
- `getStaticPaths` 的 `fallback: true` 或 `'blocking'`

因此语言必须作为**普通动态路由参数** `[locale]`，并在构建时**枚举全部**页面。`next-i18next` 官方 Pages Router SSG 示例也采用 `[locale]` 目录，且**不把 `i18n` 配置传给 `next.config.ts`**——本文档主流程不依赖 `next-i18next`，但路由设计沿用这一约定。

> 页面/组件从 `react-i18next` 导入 `useTranslation`、`Trans` 等；`_app.tsx` 用 `useMemo` + `I18nextProvider` 同步初始化 i18next 实例。不再有 `next-i18next/pages` 的子路径导入。

---

## 一、最终路由设计

```text
/                       静态语言选择页
/en/                    英文首页
/zh-CN/                 中文首页
/en/<page>/             英文子页
/zh-CN/<page>/          中文子页
```

构建产物形如 `out/{en,zh-CN}/<page>/index.html`。`/api/...` 路由仍由 Worker 处理，不参与 i18n——`api` 不是受支持语言，不会与 `[locale]` 路由冲突。

## 四、统一语言配置

`settings.json` 是单一来源；`settings.ts` 派生类型与校验函数。为拿到字面量联合类型，`settings.ts` 再维护一份 `const LOCALES`——**模块加载时校验它和 `settings.json` 一致**，不一致直接 `throw`，避免双源漂移（review 反复指出 `as` 断言无运行时校验会静默出错）。

### `src/i18n/settings.json`

```json
{
  "defaultLocale": "en",
  "locales": ["en", "zh-CN"]
}
```

### `src/i18n/settings.ts`

```ts
import settings from "./settings.json";

// 字面量联合，让 AppLocale 收紧为 "en" | "zh-CN"。
// LOCALES 与 settings.json 的 locales 是双源，模块加载时校验一致。
const LOCALES = ["en", "zh-CN"] as const;

if (
  settings.locales.length !== LOCALES.length ||
  !settings.locales.every((l, i) => l === LOCALES[i])
) {
  throw new Error("i18n/settings.json locales does not match LOCALES const");
}
// 用 (LOCALES as readonly string[]).includes 而非 as 断言,避免把未知值伪装成字面量联合。
if (!(LOCALES as readonly string[]).includes(settings.defaultLocale)) {
  throw new Error("defaultLocale not in LOCALES");
}

export const DEFAULT_LOCALE = settings.defaultLocale as (typeof LOCALES)[number];
export const SUPPORTED_LOCALES = LOCALES;
export type AppLocale = (typeof LOCALES)[number];

// localStorage key:LanguageSwitcher 写入,根路径页读取。抽常量避免双源,改名漏改一处会
// 静默失效(语言选择页不再自动跳转到上次选择的语言,无报错)。
export const PREFERRED_LOCALE_KEY = "preferred-locale";

export function isSupportedLocale(value: unknown): value is AppLocale {
  return typeof value === "string" && (LOCALES as readonly string[]).includes(value);
}

export function normalizeLocale(value: string | undefined | null): AppLocale {
  if (!value) return DEFAULT_LOCALE;
  if (isSupportedLocale(value)) return value;
  const language = value.split("-")[0];
  return LOCALES.find((l) => l.split("-")[0] === language) ?? DEFAULT_LOCALE;
}

// pickLocale 输出的文案结构:递归,key 对应 string 或嵌套子对象。
// 用递归类型而非 Record<string, unknown>,因为 unknown 不能赋值给 i18next 的
// ResourceKey(string | { [key: string]: ResourceKey }),TS 严格模式会报编译错误。
// _app.tsx 的 resources 和 i18n-static.ts 的 I18nProps.messages 都用这个类型。
export type Messages = Record<string, string | Messages>;
```

> 不导出 `getDirection`：本项目只支持 LTR 语言。真正加 RTL 语言时再补，并配套镜像图标、`rtl:` Tailwind 变体——单独设 `<Html dir>` 而无配套样式没意义。
> 项目 `tsconfig.json` 已启用 `resolveJsonModule` 和 `esModuleInterop`，无需改动。

## 六、`_app.tsx`

`_app.tsx` 用 `useMemo` 同步创建 i18next 实例 + `I18nextProvider` 注入——**不要用 `useEffect` 异步注入**,SSG 构建期 `renderToStaticMarkup` 会跳过 `useEffect`,组件渲染时资源为空,`t("todo:title")` 返回 key 本身,静态 HTML 含翻译 key 而非文案,SEO 抓取到 key 字符串。保留现有 `Layout`、warmup 等逻辑,**根路径页(`router.pathname === "/"`)绕过 `Layout`** 避免用户在语言选择页看到 Sidebar:

```tsx
import "@/styles/globals.css";
import { useEffect, useMemo } from "react";
import Head from "next/head";
import type { AppProps } from "next/app";
import { useRouter } from "next/router";
import i18next from "i18next";
import { I18nextProvider, initReactI18next, type i18n as I18nInstance } from "react-i18next";
import { Layout } from "@/components/layout/layout";
import { warmupNetworkPermission } from "@/lib/api";
import { DEFAULT_LOCALE, SUPPORTED_LOCALES, type AppLocale, type Messages } from "@/i18n/settings";

// pageProps 来自第九节 getI18nProps:locale 已收紧为 AppLocale,
// messages 是当前 locale 的 pickLocale 结果(嵌套对象,ns → key → string | 嵌套),
// fallbackMessages 是 DEFAULT_LOCALE 的 pickLocale 结果(用于 i18next fallbackLng 回退)。
type PageProps = {
  locale: AppLocale;
  // 用递归类型 Messages 反映 pickLocale 后的嵌套结构(ns → key → string | 嵌套),
  // 且类型可赋值给 i18next 的 ResourceKey(string | { [key: string]: ResourceKey })。
  // `unknown` 不可赋值给 ResourceKey,TS 严格模式会报编译错误。
  messages: Messages;
  fallbackMessages: Messages;
};

// AppProps<PageProps> 只覆盖 i18n 相关 props(locale/messages/fallbackMessages)。
// 页面若有 `data` 等额外 props,由 getStaticProps 直接调用 getI18nProps 合并(见第九节
// makeStaticProps 注释),`<Component {...pageProps} />` 把全部 pageProps 透传给页面,
// 页面组件自己声明 `data` 等额外 prop 类型——`App` 不约束页面扩展 props。
function App({ Component, pageProps }: AppProps<PageProps>) {
  const router = useRouter();
  const isLanguageLanding = router.pathname === "/";

  useEffect(() => {
    void warmupNetworkPermission();
  }, []);

  // 同步初始化:SSG 构建期 useMemo 会执行,资源在 render 前就位,
  // 避免 useEffect 异步注入导致静态 HTML 含翻译 key 而非文案。
  const i18n = useMemo<I18nInstance>(() => {
    const instance = i18next.createInstance();
    instance.use(initReactI18next).init({
      lng: pageProps.locale,
      defaultNS: "common",
      supportedLngs: [...SUPPORTED_LOCALES],
      react: { useSuspense: false },
      // 同时加载当前 locale 和 DEFAULT_LOCALE 资源,让 fallbackLng 真正生效——
      // 当前 locale 缺 key 时 i18next 回退到 DEFAULT_LOCALE 找。
      // getI18nProps 总是设置 fallbackMessages(locale === DEFAULT_LOCALE 时与 messages 同引用),
      // 无需 ?? 兜底。类型 Messages 可赋值给 i18next 的 ResourceKey。
      resources: {
        [pageProps.locale]: pageProps.messages,
        [DEFAULT_LOCALE]: pageProps.fallbackMessages,
      },
      fallbackLng: DEFAULT_LOCALE,
      interpolation: { escapeValue: false },
    });
    return instance;
  }, [pageProps.locale, pageProps.messages, pageProps.fallbackMessages]);

  // useMemo 重新计算时(pageProps.messages 引用变化),旧 i18next 实例未 dispose
  // 会泄漏事件监听器/资源 bundle。在 i18n 引用变化时清理。
  useEffect(() => {
    return () => {
      i18n.dispose?.();
    };
  }, [i18n]);

  const tree = (
    <I18nextProvider i18n={i18n}>
      <Component {...pageProps} />
    </I18nextProvider>
  );

  if (isLanguageLanding) {
    return (
      <>
        <Head>
          {/* 保留现有 viewport meta */}
          <meta
            name="viewport"
            content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no, viewport-fit=cover"
          />
        </Head>
        {tree}
      </>
    );
  }

  return (
    <>
      <Head>
        {/* 保留现有 viewport meta */}
        <meta
          name="viewport"
          content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no, viewport-fit=cover"
        />
      </Head>
      <Layout>{tree}</Layout>
    </>
  );
}

export default App;
```

> SSG 构建期 `router.pathname` 已确定（根路径页产物是 `/`），无需担心 hydration 闪烁——`isLanguageLanding` 在 SSG 产物里就是 `true`。

## 七、`_document.tsx`

`<Html lang>` 只能在构建时一次性确定。**不要用 `this as any` 读 `__NEXT_DATA__`**，用 class 写法 + `getInitialProps(ctx)`，从 `ctx.query.locale` 读 locale（动态路由参数会出现在 query 里）：

```tsx
import Document, {
  Html,
  Head,
  Main,
  NextScript,
  type DocumentContext,
  type DocumentInitialProps,
} from "next/document";
import { DEFAULT_LOCALE, isSupportedLocale, type AppLocale } from "@/i18n/settings";

type AppDocumentProps = DocumentInitialProps & { locale: AppLocale };

export default class AppDocument extends Document<AppDocumentProps> {
  static async getInitialProps(ctx: DocumentContext): Promise<AppDocumentProps> {
    const initialProps = await Document.getInitialProps(ctx);
    const queryLocale = ctx.query.locale;
    const locale =
      typeof queryLocale === "string" && isSupportedLocale(queryLocale)
        ? queryLocale
        : DEFAULT_LOCALE;
    return { ...initialProps, locale };
  }

  render() {
    const { locale } = this.props;
    return (
      <Html lang={locale}>
        <Head />
        <body className="antialiased">
          <Main />
          <NextScript />
        </body>
      </Html>
    );
  }
}
```

> 用 class 写法而非 `Document.getInitialProps = ...` 静态属性赋值，避免 TS 严格模式下「read-only」报错。`Document<AppDocumentProps>` 泛型让 `this.props` 收紧为含 `locale`，无需 `as` 断言。

## 八、客户端 i18n 工具

`src/lib/i18n-utils.ts`：路径工具，**客户端安全**（不依赖 `serverSideTranslations`），被 `LocalizedLink`、`LanguageSwitcher`、`Seo`、`isActive` 共用。**不要放进 `lib/i18n-static.ts`**——后者依赖 `serverSideTranslations`，浏览器组件不能导入。

```ts
import { isSupportedLocale, type AppLocale } from "@/i18n/settings";

// URL base 是内部实现细节,4 个函数(localizePath/stripLocalePrefix/createLocalizedUrl
// 以及 navigate.tsx 的 isActive)共用。抽常量避免 magic string 重复,改一处漏一处会静默出错。
export const LOCAL_URL_BASE = "https://local.invalid";

function isExternalHref(href: string): boolean {
  // 含协议相对 URL(//example.com/...)和锚点,避免 //example.com 被当内部路径注入 locale。
  return /^(https?:|mailto:|tel:|#|\/\/)/.test(href);
}

/**
 * 把语言注入到 href 的第一段。
 * 第一段已是受支持语言则替换之；否则插入到最前。外部链接原样返回。
 * 空字符串视为根路径。输出统一带尾斜杠,与 next.config.ts 的 trailingSlash: true 一致,
 * 避免 <Link href="/en"> 点击后多一次 redirect 到 /en/。保留 search 和 hash。
 *
 * locale 参数收紧为 AppLocale,调用方传非支持语言时 TS 报错,避免运行时产出
 * /ja/todo/ 后 404。调用方从 router.query.locale 拿到值后用 isSupportedLocale 收紧再传入。
 */
export function localizePath(href: string, locale: AppLocale): string {
  if (!href) return `/${locale}/`;
  if (isExternalHref(href)) return href;

  const url = new URL(href.startsWith("/") ? href : `/${href}`, LOCAL_URL_BASE);
  const segments = url.pathname.split("/").filter(Boolean);

  if (segments.length > 0 && isSupportedLocale(segments[0])) {
    segments[0] = locale;
  } else {
    segments.unshift(locale);
  }

  return `/${segments.join("/")}/${url.search}${url.hash}`;
}

/** 移除路径里的语言前缀，返回不含语言的裸路径。用于 Seo 拼接 canonical/hreflang。 */
export function stripLocalePrefix(path: string): string {
  if (isExternalHref(path)) return path;
  const url = new URL(path.startsWith("/") ? path : `/${path}`, LOCAL_URL_BASE);
  const segments = url.pathname.split("/").filter(Boolean);
  if (segments.length > 0 && isSupportedLocale(segments[0])) segments.shift();
  const pathname = segments.length === 0 ? "/" : `/${segments.join("/")}`;
  return `${pathname}${url.search}${url.hash}`;
}

/** 替换当前路径的语言段，保留查询参数和 hash。与 localizePath 等价。 */
export function replacePathLocale(currentPath: string, nextLocale: AppLocale): string {
  return localizePath(currentPath, nextLocale);
}

/**
 * 获取站点 URL,生产构建时未设 NEXT_PUBLIC_SITE_URL 直接 throw。
 * Seo 和根路径页共用,避免双源(根路径页硬编码 example.com 会与 Seo 不一致)。
 */
export function getSiteUrl(): string {
  const url = process.env.NEXT_PUBLIC_SITE_URL;
  if (!url && process.env.NODE_ENV === "production") {
    throw new Error("NEXT_PUBLIC_SITE_URL must be set for production build");
  }
  return url ?? "https://example.com";
}
```

## 九、静态 i18n 工具

`src/lib/i18n-static.ts` **只能在 `getStaticProps` 中使用**，不要从浏览器组件导入。

```ts
import type { GetStaticPaths, GetStaticPropsContext, GetStaticPropsResult } from "next";
import { DEFAULT_LOCALE, isSupportedLocale, SUPPORTED_LOCALES, type AppLocale, type Messages } from "@/i18n/settings";
// 合并格式 JSON:每 key 带 { en, "zh-CN" } 语言对象。@/* 别名只到 ./src/*,翻译文件在
// public/locales/,用相对路径。每新增 namespace 同步 import 并加入 MESSAGES 映射。
import common from "../../public/locales/common.json";
import todo from "../../public/locales/todo.json";
import test from "../../public/locales/test.json";

type MergedMessages = Record<string, unknown>;
const MESSAGES: Record<string, MergedMessages> = {
  common,
  todo,
  test,
};

export type StaticPageContext = GetStaticPropsContext<{
  locale?: string;
  [key: string]: string | string[] | undefined;
}>;

// locale 收紧为 AppLocale,让 narrowing 一路传到页面,页面 props.locale 可直接传给
// 期望 AppLocale 的函数(如 localizePath)。messages/fallbackMessages 用递归类型 Messages
// (第四节定义),反映 pickLocale 后的嵌套结构且可赋值给 i18next 的 ResourceKey。
export type I18nProps = {
  locale: AppLocale;
  messages: Messages;
  fallbackMessages: Messages;
};

/**
 * 把合并格式的 JSON({ key: { en: "...", "zh-CN": "..." } })按当前 locale 提取
 * 出该 locale 的文案。在加载文件的地方(getI18nProps)转换,把「按 locale 分目录」
 * 变成「文件内按 key 内联」,新增语言只改 JSON 不动目录。
 *
 * **leaf 判定**:对象的所有 key 都 ∈ SUPPORTED_LOCALES 且 value 是 string。中间节点递归。
 * 边界:language.en 的 value { en: "English", "zh-CN": "English" } 也是 leaf,正确提取。
 *
 * **缺字段回退**:leaf 缺当前 locale 字段(如 { en: "Home" } 缺 zh-CN)时 `node[locale]` 是
 * undefined。调用方(getI18nProps)同时 pickLocale DEFAULT_LOCALE 作为 fallbackMessages,
 * _app.tsx 的 i18next 配 fallbackLng: DEFAULT_LOCALE + 双 locale resources,让 i18next
 * 在当前 locale 拿到 undefined 时回退到 DEFAULT_LOCALE。
 *
 * **类型注记**:返回类型签名是 Messages(不含 undefined),但 `isLeaf` 分支的
 * `(node as Record<string, string>)[locale] as unknown as Messages` cast 隐藏了 undefined
 * (leaf 缺字段时该 key 的 value 运行时是 undefined)。调用方应通过 `t()` 访问文案
 * (i18next 内部处理 undefined 为缺失走 fallbackLng),不要直接读 `messages[ns].key`
 * (绕过 t() 不会触发 fallback,可能拿到 undefined)。同理 primitive/array 分支的
 * `as unknown as Messages` cast 也是 type-unsound,但正常调用不走到(防御性代码)。
 *
 * **边界**:leaf 误加非支持语言的 key(如 { en: "Home", fr: "Maison" })时 isLeaf = false,
 * 递归处理,value 是 string 原样返回,最终 messages[ns].key 是对象而非 string,t() 拿不到
 * 文案。第十节明确禁止 leaf 含 non-locale key。
 */
export function pickLocale(node: unknown, locale: AppLocale): Messages {
  // primitive/array 分支:防御性代码,正常调用不走到这里。cast 是 pragmatic。
  if (node === null || typeof node !== "object") return node as unknown as Messages;
  if (Array.isArray(node)) return node.map((item) => pickLocale(item, locale)) as unknown as Messages;

  const entries = Object.entries(node as Record<string, unknown>);
  const isLeaf =
    entries.length > 0 &&
    entries.every(
      ([k, v]) =>
        (SUPPORTED_LOCALES as readonly string[]).includes(k) && typeof v === "string",
    );

  // leaf 缺当前 locale 字段时 node[locale] 是 undefined,as unknown as Messages 隐藏了它
  // (string 与 Messages 不可比较,需双重断言)。调用方应通过 t() 访问,走 fallbackLng。
  if (isLeaf) return (node as Record<string, string>)[locale] as unknown as Messages;

  const result: Messages = {};
  for (const [k, v] of entries) result[k] = pickLocale(v, locale);
  return result;
}

export function getLocaleFromContext(ctx: StaticPageContext): AppLocale | null {
  const locale = ctx.params?.locale;
  return isSupportedLocale(locale) ? locale : null;
}

/** 用于只有 [locale] 一个动态参数的页面。 */
export const getLocaleStaticPaths: GetStaticPaths = async () => ({
  paths: SUPPORTED_LOCALES.map((locale) => ({ params: { locale } })),
  fallback: false, // output: 'export' 必须预生成全部路径
});

export async function getI18nProps(
  ctx: StaticPageContext,
  namespaces: string[],
): Promise<I18nProps | null> {
  const locale = getLocaleFromContext(ctx);
  if (!locale) return null;

  // 直接 import 合并格式 JSON + pickLocale 转换,不依赖 serverSideTranslations。
  // 后者按 localePath/<locale>/<ns>.json 读文件,与合并格式不兼容。
  // 同时 pickLocale DEFAULT_LOCALE 作为 fallbackMessages,让 _app.tsx 的 fallbackLng
  // 真正生效——当前 locale 漏某 leaf 字段时 i18next 回退到 DEFAULT_LOCALE 找。
  // locale === DEFAULT_LOCALE 时复用同一引用,避免 useMemo 因引用不同触发无意义重算。
  const messages: Messages = {};
  const fallbackMessages: Messages = {};
  for (const ns of namespaces) {
    const merged = MESSAGES[ns];
    if (!merged) continue; // namespace 未注册到 MESSAGES,messages[ns] 不设置
    // → t("newNs:key") 在客户端找不到资源(ns 级别缺失,fallbackLng 无法补救,返回 key 本身)。
    const picked = pickLocale(merged, locale);
    messages[ns] = picked;
    fallbackMessages[ns] = locale === DEFAULT_LOCALE ? picked : pickLocale(merged, DEFAULT_LOCALE);
  }

  return { locale, messages, fallbackMessages };
}

/**
 * 不需要额外数据的页面用这个 helper。页面需同时取数据时直接写 getStaticProps，
 * 调用 getI18nProps 并合并 data，不要走这个 helper。
 *   const i18nProps = await getI18nProps(ctx, ["common", "todo"]);
 *   if (!i18nProps) return { notFound: true as const };
 *   const data = await fetchSomeData(i18nProps.locale);
 *   return { props: { ...i18nProps, data } };
 */
export function makeStaticProps(namespaces: string[]) {
  return async (ctx: StaticPageContext): Promise<GetStaticPropsResult<I18nProps>> => {
    const props = await getI18nProps(ctx, namespaces);
    if (!props) return { notFound: true as const };
    return { props };
  };
}

export { DEFAULT_LOCALE, SUPPORTED_LOCALES };
```

> `makeStaticProps` 不设默认 namespaces——强制调用方传，避免漏 namespace。

## 十、翻译文件

### 目录结构

不再按 locale 分目录,所有 namespace 文件平铺在 `public/locales/` 下,每 key 内联所有语言:

```text
public/
└── locales/
    ├── common.json           全局:导航、按钮、通用错误、语言名、siteName
    ├── todo.json             todo 页:meta、toast、字段标签、空状态
    ├── test.json             test 页:meta、placeholder、空状态
    └── <page>.json           每个带语言的页面一个专属 namespace
```

每新增 namespace,在 `src/lib/i18n-static.ts` 的 `MESSAGES` 映射里同步 import + 加入(第九节)。

### 文件格式:合并语言对象

每个文案 key 的 value 是 `{ [locale]: string }` 的对象,leaf 节点的 key 集合**只能**是 `SUPPORTED_LOCALES` 的子集(不能加 `fr` 等非支持语言 key,否则 `pickLocale` 的 `isLeaf` 判定失败,递归后返回对象而非 string,`t()` 拿不到文案),所有 value 是 string。中间节点是「业务 key → 子对象」,递归。

```json
// public/locales/common.json
{
  "siteName": { "en": "Example", "zh-CN": "示例" },
  "navigation": {
    "home": { "en": "Home", "zh-CN": "首页" },
    "todo": { "en": "Todo", "zh-CN": "待办" },
    "test": { "en": "Test", "zh-CN": "测试" }
  },
  "language": {
    "label": { "en": "Language", "zh-CN": "语言" },
    "en":     { "en": "English",   "zh-CN": "English" },
    "zh-CN":  { "en": "简体中文",   "zh-CN": "简体中文" }
  },
  "error": {
    "requestFailed": { "en": "Request failed", "zh-CN": "请求失败" }
  }
}
```

```json
// public/locales/todo.json
{
  "metaTitle":       { "en": "Todo | Example",       "zh-CN": "待办 | 示例" },
  "metaDescription": { "en": "Manage your tasks.",   "zh-CN": "管理你的任务。" },
  "title":           { "en": "Todo",                 "zh-CN": "待办" },
  "addPlaceholder":  { "en": "What needs to be done?", "zh-CN": "要做什么?" },
  "empty":           { "en": "Nothing here yet.",   "zh-CN": "还没有内容。" },
  "toast": {
    "added":     { "en": "Added",       "zh-CN": "已添加" },
    "addFailed": { "en": "Add failed",  "zh-CN": "添加失败" },
    "saved":     { "en": "Saved",       "zh-CN": "已保存" },
    "deleted":   { "en": "Deleted",     "zh-CN": "已删除" }
  }
}
```

### 加载时 `pickLocale` 转换

`getI18nProps`(第九节)import 合并 JSON 后,用 `pickLocale(merged, locale)` 提取当前 locale 的文案(嵌套对象,保留 `navigation.home` 这种层级结构),组装成 `{ [ns]: 嵌套对象 }` 给 `_app.tsx` 的 `i18next.init({ resources: { [locale]: messages, [DEFAULT_LOCALE]: fallbackMessages } })`。`pickLocale` 的 leaf 判定:对象所有 key ∈ `SUPPORTED_LOCALES` 且 value 是 string,如 `{ "en": "Home", "zh-CN": "首页" }`;中间节点递归。`language.en` 这种 key 名与 locale 撞的场景也正确处理(value 是 leaf 对象)。

### namespace 组织原则

- **一个 namespace = 一个 JSON 文件**,文件名即 namespace 名(`todo.json` → `useTranslation(["todo"])` → `t("todo:key")`)。
- **`common` 是全局 namespace**,放所有页面共用的文案:导航、通用按钮、通用错误、语言名/标签、SEO 站名。
- **每个页面一个专属 namespace**,文件名与路由名对应:`src/pages/[locale]/todo.tsx` → `public/locales/todo.json`。
- **页面只加载自己需要的 namespace**:`useTranslation(["common", "todo"])` 不会加载 `test.json`,减体积。
- **缺 key 回退**:某 key 在当前 locale 的合并对象里漏了对应语言字段(如 `{ "en": "Home" }` 缺 `"zh-CN"`),`pickLocale` 返回 `undefined`。`getI18nProps` 同时 `pickLocale(merged, DEFAULT_LOCALE)` 作为 `fallbackMessages`,`_app.tsx` 的 i18next 配 `fallbackLng: DEFAULT_LOCALE` + 双 locale `resources`,让 `t()` 在当前 locale 拿到 `undefined` 时回退到 `DEFAULT_LOCALE` 找文案。若 `DEFAULT_LOCALE` 也缺该 key,仍返回 key 本身。

### key 命名约定

- **小驼峰**:`metaTitle`、`addToCart`、`loadFailed`。
- **嵌套用对象,不用点号 key**:`{ "toast": { "added": { "en": "Added", "zh-CN": "已添加" } } }` 而非 `{ "toast.added": ... }`——i18next 解析 `t("toast.added")` 时按 `.` 分层进对象。
- **业务域前缀分组**:`navigation.*`(导航)、`toast.*`(操作反馈)、`error.*`(错误)、`field.*`(表单字段)、`meta.*`(SEO meta)。

> `siteName` 用于 `<title>` 等 SEO 字段,与页内品牌名是两件事。品牌名通常不翻译("Apple" 在所有语言下都是 "Apple"),不要塞进 `common.json`。

### 新增语言的完整步骤

1. `src/i18n/settings.json` 的 `locales` 数组加新 locale(如 `"ja"`)。
2. `src/i18n/settings.ts` 的 `LOCALES` const 同步加(模块加载时运行时校验会强制二者一致,不一致直接 throw)。
3. 在每个 `public/locales/*.json` 文件里,给每个 leaf 对象补 `"ja": "<日语文案>"`——**只改文件、不加目录**。可写脚本扫描所有 leaf 并补字段。
4. `common.json` 的 `language` 对象加 `"ja": { "en": "日本語", "ja": "日本語", "zh-CN": "日本語" }`(每个 locale 下如何称呼日语)。
5. 根路径页 `labels`(第十七节)同步加 `"ja": "日本語"`,或抽到 `src/i18n/language-labels.json` 共享(避免 triplication)。
6. `404.tsx` 的 `messages`(第十八节)同步加 `ja` 条目。
7. 重新 `pnpm build` 静态导出,验证 `out/ja/` 下所有页面生成且文案正确。

合并格式相比按 locale 分目录的优势:新增语言只改 JSON 文件内容,不复制目录结构;`pickLocale` 在加载层统一转换,调用方(页面/组件)无感知。

## 十一、页面通用结构

所有 `[locale]/*` 页面统一结构：`getStaticPaths = getLocaleStaticPaths`；`getStaticProps = makeStaticProps([...namespaces])` 或直接调用 `getI18nProps` 合并业务数据；组件内 `const { t } = useTranslation([...namespaces])`（统一用数组形式）。最小模板：

```tsx
import { useTranslation } from "react-i18next";
import { Seo } from "@/components/i18n/Seo";
import { getLocaleStaticPaths, makeStaticProps } from "@/lib/i18n-static";

export default function TodoPage() {
  const { t } = useTranslation(["common", "todo"]);
  return (
    <>
      <Seo title={t("todo:metaTitle")} description={t("todo:metaDescription")} path="/todo" />
      <main>{/* 页面内容 */}</main>
    </>
  );
}

export const getStaticPaths = getLocaleStaticPaths;
export const getStaticProps = makeStaticProps(["common", "todo"]);
```

页面需同时取业务数据时，不要用 `makeStaticProps`，直接调用 `getI18nProps` 合并 `data`（见第九节注释）。

`<Seo>` 的 `title` 和 `description` 都走 i18n key（`t("pageNs:metaTitle")` / `t("pageNs:metaDescription")`），**不要硬编码**，也不要把页面标题当 description 用（太短，对 SEO 不友好）。每个页面 namespace 应有独立的 `metaTitle` 和 `metaDescription` key。

## 十二、Layout 与导航通用迁移

迁移要点（**不列具体页面代码**，原则照此套用）：

1. **`navItems.href` 保持不带语言前缀**。`LocalizedLink` 在运行时根据 `router.query.locale` 注入前缀，避免在 `navItems` 里维护多语言两份。
2. **`navItems.title` 改 i18n key**，渲染时 `t(item.titleKey)` 解析。
3. **`Sidebar` / `Tabbar` 内部所有 `<Link>` 替换为 `LocalizedLink`**；`onPointerDown` 里的 `router.push(item.href)` 也要改成 `router.push(localizePath(item.href, currentLocale))`。`currentLocale` 从 `router.query.locale` 读，用 `isSupportedLocale` 校验后 fallback 到 `DEFAULT_LOCALE`：

   ```ts
   const currentLocale =
     typeof router.query.locale === "string" && isSupportedLocale(router.query.locale)
       ? router.query.locale
       : DEFAULT_LOCALE;
   ```

   `SidebarHeader` 品牌名不翻译。
4. **`isActive` 必须重写**：`useRouter().pathname` 返回的是路由模板（如 `/[locale]/todo`），不是实际 URL，直接和 `item.href` 比永远不等。改用 `useRouter().asPath` + `stripLocalePrefix`：

   ```ts
   import { LOCAL_URL_BASE, stripLocalePrefix } from "@/lib/i18n-utils";

   function isActive(asPath: string, href: string) {
     // 只比 pathname,忽略 search/hash——否则 /zh-CN/todo?page=2 strip 后是 /todo?page=2,
     // 与 /todo/ 不等,nav 项不高亮。stripLocalePrefix 保留 search/hash 是为 Seo 拼接 canonical
     // 设计的,isActive 这里要先取 pathname 再 strip。LOCAL_URL_BASE 与 i18n-utils 共用。
     const path = new URL(asPath, LOCAL_URL_BASE).pathname;
     return (
       stripLocalePrefix(path).replace(/\/$/, "") === href.replace(/\/$/, "")
     );
   }
   ```

5. **`LanguageSwitcher` 位置**：放进 `src/components/layout/navigate.tsx` 的 `Sidebar` 组件的 `<SidebarFooter>` 内、`<SidebarTrigger>` 上方。移动端 `Tabbar` 底部空间紧张不放——落地时二选一短期补救（移动端 `Layout` 顶部加紧凑 switcher，或用 shadcn `Sheet`/`Drawer` 承载），**不要留「后续再补」**。

## 十三、既有页面迁移通用流程

1. **扫描硬编码字符串**：`grep` 出所有中文/英文文案（包括 `toast.success("...")`、placeholder、错误 fallback、品牌名外的 UI 文本）。
2. **按 namespace 归类**：通用文案进 `common.json`，页面专属进 `<page>.json`。
3. **抽到 JSON**：每个字符串一个 key，按业务域组织（如 `toast.added`、`error.requestFailed`）。
4. **组件替换**：`<Button>Primary</Button>` → `<Button>{t("pageNs:button.primary")}</Button>`；`toast.success("已添加")` → `toast.success(t("pageNs:toast.added"))`。
5. **工具模块特殊处理**：`src/lib/api.ts` 等不在 React 树里的模块，`parseError` 不再 fallback 到中文字符串，改为抛带 code 的 `ApiError`，由调用方在组件 catch 里用 `e.message || t("common:error.requestFailed")` 兜底。

   `ApiError` 由项目按自身约定设计，本文档只规定接口：至少有 `code`（字符串，用于前端按 code 映射本地化文案，可测）和 `message`（后端返回的原文案，可能与当前 locale 无关）两个字段，可选 `status`（HTTP 状态码）。最小定义示例：

   ```ts
   export class ApiError extends Error {
     constructor(
       public code: string,
       public status?: number,
       message?: string,
     ) {
       super(message);
       this.name = "ApiError";
     }
   }
   ```

   调用方 catch 时按需区分：`e instanceof ApiError` 时用 `e.code` 走 i18n 映射（推荐，可测），用 `e.message` 兜底（接受后端原文案不翻译的妥协）；非 `ApiError` 的未知错误走 `t("common:error.requestFailed")`。
6. **后端错误文案策略**：`e.message` 是后端返回的文案，与当前 locale 无关。要么后端返回 error code 由前端映射到本地化文案（推荐，可测），要么接受后端文案不翻译（可能出现「中文页面显示英文后端错误」，是已知妥协）。

> 用字符串本身定位硬编码，**不要用行号**——任何源文件编辑都会让行号漂移。

## 十四、`LocalizedLink`

```tsx
import Link from "next/link";
import type { ComponentProps, ReactNode } from "react";
import { useRouter } from "next/router";
import { DEFAULT_LOCALE, isSupportedLocale, type AppLocale } from "@/i18n/settings";
import { localizePath } from "@/lib/i18n-utils";

// locale prop 收紧为 AppLocale,<LocalizedLink href="/todo" locale="ja"> 编译期报错,
// 不会运行时产出 /ja/todo/ 后 404。
type Props = Omit<ComponentProps<typeof Link>, "href"> & {
  href: string;
  locale?: AppLocale;
  children: ReactNode;
};

export function LocalizedLink({ href, locale: requestedLocale, children, ...props }: Props) {
  const router = useRouter();
  const routeLocale =
    typeof router.query.locale === "string" && isSupportedLocale(router.query.locale)
      ? router.query.locale
      : DEFAULT_LOCALE;
  const locale = requestedLocale ?? routeLocale;
  return (
    <Link href={localizePath(href, locale)} {...props}>
      {children}
    </Link>
  );
}
```

用法：`<LocalizedLink href="/todo">Todo</LocalizedLink>` 在 `/zh-CN/` 下渲染为 `/zh-CN/todo`；传 `locale="en"` 强制渲染为 `/en/todo`。

## 十五、`LanguageSwitcher`

切换语言时保留当前页面、查询参数和 hash（`/zh-CN/todo?page=2#specs` → `/en/todo?page=2#specs`），并写 `localStorage["preferred-locale"]` 供根路径页读取。**URL 是当前语言的唯一来源**——不要只调 `i18n.changeLanguage()`，否则 URL 不变、刷新后回到原语言、canonical/hreflang 与页面内容不一致。

```tsx
import { useRouter } from "next/router";
import { useTranslation } from "react-i18next";
import { DEFAULT_LOCALE, isSupportedLocale, SUPPORTED_LOCALES, PREFERRED_LOCALE_KEY } from "@/i18n/settings";
import { replacePathLocale } from "@/lib/i18n-utils";

export function LanguageSwitcher() {
  const router = useRouter();
  const { t } = useTranslation(["common"]);
  const currentLocale =
    typeof router.query.locale === "string" && isSupportedLocale(router.query.locale)
      ? router.query.locale
      : DEFAULT_LOCALE;

  // nextLocale 来自 <select>.value,运行时是 string,入口用 isSupportedLocale 收紧再继续,
  // 防止 option 列表被污染或调试时手动改值写入非法 locale。
  async function changeLocale(nextLocale: string) {
    if (!isSupportedLocale(nextLocale)) return;
    if (nextLocale === currentLocale) return;
    localStorage.setItem(PREFERRED_LOCALE_KEY, nextLocale);
    await router.push(replacePathLocale(router.asPath, nextLocale));
  }

  return (
    <label>
      <span className="sr-only">{t("common:language.label")}</span>
      <select
        value={currentLocale}
        onChange={(e) => void changeLocale(e.target.value)}
      >
        {SUPPORTED_LOCALES.map((l) => (
          <option key={l} value={l}>{t(`common:language.${l}`)}</option>
        ))}
      </select>
    </label>
  );
}
```

## 十六、SEO：canonical 与 hreflang

> **适用范围**：`Seo.tsx` 仅用于 `src/pages/[locale]/*` 下页面（这些页面有 `router.query.locale`）。根路径语言选择页和 `404.tsx` 各自维护 `<Head>`，因为它们的 `<head>` 不依赖当前 locale。

```tsx
import Head from "next/head";
import { useRouter } from "next/router";
import { DEFAULT_LOCALE, isSupportedLocale, SUPPORTED_LOCALES, type AppLocale } from "@/i18n/settings";
import { getSiteUrl, LOCAL_URL_BASE, stripLocalePrefix } from "@/lib/i18n-utils";

type SeoProps = {
  title: string;
  description: string;
  /** 不含语言前缀的路径。首页传 "/"，todo 传 "/todo"。误传 "/en/todo" 也会被 stripLocalePrefix 规整。 */
  path: string;
};

// 复用第八节 getSiteUrl() 的 fail-loud 逻辑,避免与根路径页两处独立维护。
const SITE_URL = getSiteUrl();

function createLocalizedUrl(locale: AppLocale, path: string) {
  // canonical/hreflang 只用 pathname,不带 search/hash。
  // stripLocalePrefix 保留 search/hash,这里用 URL 解析只取 pathname 部分,
  // 避免 /todo?page=2 被拼成 /en/todo?page=2/(查询参数跑到路径中间)。
  // stripLocalePrefix 返回值总是以 `/` 开头(pathname 是 `/` 或 `/segments`),无需再补前缀。
  const stripped = stripLocalePrefix(path);
  const url = new URL(stripped, LOCAL_URL_BASE);
  const segments = url.pathname.split("/").filter(Boolean);
  const pathname = segments.length === 0 ? "" : `/${segments.join("/")}`;
  return `${SITE_URL}/${locale}${pathname}/`;
}

export function Seo({ title, description, path }: SeoProps) {
  const router = useRouter();
  const locale =
    typeof router.query.locale === "string" && isSupportedLocale(router.query.locale)
      ? router.query.locale
      : DEFAULT_LOCALE;
  return (
    <Head>
      <title>{title}</title>
      <meta name="description" content={description} />
      <link rel="canonical" href={createLocalizedUrl(locale, path)} />
      {SUPPORTED_LOCALES.map((alt) => (
        <link key={alt} rel="alternate" hrefLang={alt} href={createLocalizedUrl(alt, path)} />
      ))}
      <link rel="alternate" hrefLang="x-default" href={createLocalizedUrl(DEFAULT_LOCALE, path)} />
    </Head>
  );
}
```

页面调用示例：

```tsx
<Seo title={t("todo:metaTitle")} description={t("todo:metaDescription")} path="/todo" />
```

`path` 传不含语言前缀的裸路径（`"/todo"`、`"/"`），`stripLocalePrefix` 兜底规整；`title`/`description` 都走 i18n key。

## 十七、根路径 `/`：语言选择页

静态导出不支持 Middleware/Accept-Language 自动重定向。让 `/` 成为语言选择页，**读取 `preferred-locale` 客户端跳转**，但保留可点击选项。

**关键**：用 `sessionStorage` 标记「本次会话已自动跳转过」，避免用户从 `/zh-CN/` 后退回 `/` 时被再次自动跳走、无法停留在选择页：

```tsx
import { useEffect, useState } from "react";
import Head from "next/head";
import Link from "next/link";
import { useRouter } from "next/router";
import { DEFAULT_LOCALE, normalizeLocale, SUPPORTED_LOCALES, PREFERRED_LOCALE_KEY, type AppLocale } from "@/i18n/settings";
import { getSiteUrl } from "@/lib/i18n-utils";

// triplication 风险：labels 与 settings.json 的 locales、common.json 的 language.*
// 是三份独立维护,新增语言三处都要改。建议抽到 src/i18n/language-labels.json:
//
//   { "en": "English", "zh-CN": "简体中文" }
//
// 根路径页和 404 都 import 这份 JSON,与 common.json 的 language.* 保持同步
// (或干脆只用一份)。下面 labels 是「未抽取」的初始形态。
// 类型用 Record<AppLocale, string>,让 TS 在 SUPPORTED_LOCALES 加新语言时强制 labels 同步补全。
const labels: Record<AppLocale, string> = { en: "English", "zh-CN": "简体中文" };

const SESSION_KEY = "locale-redirected";

export default function LanguageLandingPage() {
  const router = useRouter();
  const [autoLocale, setAutoLocale] = useState<AppLocale | null>(null);

  useEffect(() => {
    // 只在首次访问（本会话未跳过）时自动跳,避免后退回 / 被再次跳走。
    if (sessionStorage.getItem(SESSION_KEY)) return;

    const stored = localStorage.getItem(PREFERRED_LOCALE_KEY);
    const detected = normalizeLocale(stored ?? navigator.language);
    setAutoLocale(detected);
    // 客户端导航:不会整页刷新,React 自然渲染"Redirecting..."。
    // 所有检测到的语言都跳转(en → /en/,zh-CN → /zh-CN/),不要只跳非默认语言——
    // 否则英文用户停在 / 需要再点一次,体验不对等。
    // sessionStorage 标记放到 replace 成功后:跳转失败时不标记,下次访问 / 仍可自动跳。
    void router.replace(`/${detected}/`).then(() => {
      sessionStorage.setItem(SESSION_KEY, "1");
    });
    // useRouter() 返回稳定引用,[] 等价于 [router];用 [] + eslint-disable 表意更清晰。
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, []);

  // 与 Seo.tsx 共用 getSiteUrl,避免 canonical URL 双源(根路径页硬编码 example.com)。
  const siteUrl = getSiteUrl();

  return (
    <>
      <Head>
        <title>Select language</title>
        <meta name="robots" content="noindex,follow" />
        <link rel="canonical" href={`${siteUrl}/${DEFAULT_LOCALE}/`} />
      </Head>
      <main>
        <h1>Select your language</h1>
        {/* 根路径页是入口，无当前 locale，用手动拼 URL；其余页面统一用 LocalizedLink。 */}
        <nav aria-label="Language selection">
          {SUPPORTED_LOCALES.map((l) => (
            <p key={l}><Link href={`/${l}/`}>{labels[l]}</Link></p>
          ))}
        </nav>
        {autoLocale && <p>Redirecting to {labels[autoLocale]}…</p>}
      </main>
    </>
  );
}
```

> `router.replace` 是客户端导航，不会立即卸载页面，React 会先渲染「Redirecting…」再切到目标路由。如果改用 `window.location.href`，setState 异步 + 立即导航会跳过 React 渲染，提示文案可能不显示。
> 根路径页的自动跳转与第二十二节第 8 条 `preferred-locale` 读写闭环：写入在 `LanguageSwitcher`，读取在这里，`sessionStorage` 防止回退死循环。

## 十八、`404.tsx`

Pages Router 的 `src/pages/404.tsx` 只会**全局**生成一个 `out/404.html`，不能按语言分别生成，**也无法用 `useTranslation`**：404 是 SSG 全局页，`_app.tsx` 的 i18next 实例由 `pageProps.locale`/`messages` 初始化，但 404 路由没有 `[locale]` 参数、`pageProps` 里也没有这两个字段。所以本地维护一份 `messages`，并在客户端从 `window.location.pathname` 推断当前语言。

`messages` 的类型用 `Record<AppLocale, ...>`（利用第四节的字面量联合），让 TS 在漏加新语言时报错，避免静默 fallback 到错误语言：

```tsx
import { useEffect, useState } from "react";
import Link from "next/link";
import { DEFAULT_LOCALE, isSupportedLocale, type AppLocale } from "@/i18n/settings";

// 维护成本：404 不能用 useTranslation,只能本地维护 messages。
// 新增语言时这里也要加一条——Record<AppLocale, ...> 让漏加时 TS 报错。
//
// 已知妥协:SSG 产物是 DEFAULT_LOCALE(en)404。非默认语言用户访问 /zh-CN/nonexistent 时,
// 先看到英文 404(静态 HTML),hydration 后 useEffect 读 window.location.pathname 切到中文,
// 有一次英文闪烁。404 不走 _document.tsx 的 ctx.query.locale 路径,无法消除。
const messages: Record<AppLocale, { title: string; description: string; home: string }> = {
  en: { title: "Page not found", description: "The requested page could not be found.", home: "Return home" },
  "zh-CN": { title: "页面不存在", description: "找不到您访问的页面。", home: "返回首页" },
};

export default function NotFoundPage() {
  const [locale, setLocale] = useState<AppLocale>(DEFAULT_LOCALE);

  useEffect(() => {
    const first = window.location.pathname.split("/").filter(Boolean)[0];
    if (isSupportedLocale(first)) setLocale(first);
  }, []);

  const m = messages[locale];
  return (
    <main>
      <h1>{m.title}</h1>
      <p>{m.description}</p>
      <Link href={`/${locale}/`}>{m.home}</Link>
    </main>
  );
}
```

不要指望 `src/pages/[locale]/404.tsx` 处理 `/zh-CN/unknown-page`——它只会生成一个普通的 `/zh-CN/404/` 页面，真正的未知路径由静态服务器的全局 404 规则处理。

## 二十一、为何不直接用 `next-i18next`

本文档主流程(第六节 `_app.tsx` 的 `useMemo` + `I18nextProvider`、第九节 `i18n-static.ts` 的 `pickLocale`、第十节合并格式文件)其实就是裸 `react-i18next` 方案,不依赖 `next-i18next`。原因:

- `output: 'export'` 下 `next-i18next` 的服务端能力几乎用不上——`serverSideTranslations` 本质就是把 JSON 打包进 props。
- 合并格式文件结构(第十节)与 `serverSideTranslations` 不兼容——后者按 `localePath/<locale>/<ns>.json` 读文件,合并格式找不到文件。
- 直接 `import` JSON + `pickLocale` 转换 + `useMemo` 同步初始化,完全可控,少一个依赖、少一份配置。

**何时考虑切回 `next-i18next` + 按 locale 分目录**:页面/namespace 数量很大、想用约定式目录扫描和 `serverSideTranslations` 自动加载、不想维护 `MESSAGES` 映射。切换时:

1. 把 `public/locales/*.json` 拆成 `public/locales/{en,zh-CN}/*.json`,每个 leaf 只保留对应 locale 的 string。
2. 删 `src/lib/i18n-static.ts` 的 `pickLocale` 和 `MESSAGES`,`getI18nProps` 改回 `serverSideTranslations(locale, namespaces, config)`。
3. `_app.tsx` 从 `useMemo` + `I18nextProvider` 改回 `appWithTranslation(App)`。
4. 新建 `next-i18next.config.ts`,设 `defaultNS`/`supportedLngs`/`fallbackLng`/`react.useSuspense`/`localePath`。

完整对比见 [react-i18next SSR 示例](https://react.i18next.com/misc/ssr) 与 [next-i18next 文档](https://github.com/i18next/next-i18next)。本项目页面少、文案少,裸 `react-i18next` + 合并格式更贴合 AGENTS.md「Simplicity First」。

## 二十二、常见错误

1. **【关键】`isActive` 用 `pathname`** — `useRouter().pathname` 返回路由模板 `/[locale]/todo`，字面 `[locale]` 匹配不到任何 locale regex，Demo/Todo/Test 三个 nav 项 active 高亮全失效。改用 `asPath` + `stripLocalePrefix`。这是迁移后最容易被用户感知的 bug。
2. **把 `i18n` 传给 `next.config.ts`** — 静态导出不支持内置 Internationalized Routing，保持 `next.config.ts` 原样。
3. **`useEffect` 异步注入 i18n 资源** — SSG 构建期 `renderToStaticMarkup` 跳过 `useEffect`，组件渲染时资源为空，`t("todo:title")` 返回 key 本身，静态 HTML 含翻译 key 而非文案。必须用 `useMemo` 同步初始化(第六节)。
4. **`fallback: true` 或 `"blocking"`** — 必须 `fallback: false`，并枚举全部 `locale × 页面参数`。
5. **只调 `i18n.changeLanguage()`** — 切换语言必须 `router.push("/zh-CN/...")`，URL 是唯一来源。
6. **在客户端组件导入 `i18n-static.ts`** — 它 `import` 合并格式 JSON + `pickLocale`，仅用于 `getStaticProps` 构建阶段。客户端只导入 `@/i18n/settings` 和 `@/lib/i18n-utils`。
7. **根路径页自动跳转死循环** — 用 `sessionStorage` 标记「本会话已跳过」，否则用户后退回 `/` 会被再次跳走（见第十七节）。
8. **新增 namespace 漏在 `MESSAGES` 注册** — `i18n-static.ts` 的 `MESSAGES` 映射漏加新 namespace,`getI18nProps` 跳过它,`t("newNs:key")` 在客户端找不到资源返回 key 本身。新增 namespace 必须同步 import + 加 `MESSAGES[ns]`。
9. **合并格式 JSON 的 leaf 漏某语言字段** — `{ "en": "Home" }` 缺 `"zh-CN"`,`pickLocale` 在 `zh-CN` 下返回 `undefined`。`getI18nProps` 同时 pickLocale `DEFAULT_LOCALE` 作为 `fallbackMessages`,`_app.tsx` 配 `fallbackLng: DEFAULT_LOCALE` + 双 locale `resources`,让 `t()` 回退到 `en` 文案。若 `en` 也缺该 key,仍返回 key 本身。新增语言时脚本扫描所有 leaf 补字段。

## 二十四、推荐原则

1. 所有**带语言**的页面放在 `src/pages/[locale]/`；根路径语言选择页除外。
2. `next.config.ts` 不配置 `i18n`。
3. 每个动态页面都枚举全部 `locale × 页面参数`。
4. 所有 `getStaticPaths()` 用 `fallback: false`。
5. 翻译文件用合并格式（`{key: {en, "zh-CN"}}`），加载时 `pickLocale` 转换；不按 locale 分目录。
6. 导航统一用 `LocalizedLink`；`navItems.href` 不带语言前缀。
7. 切换语言时同步改 URL，并写 `preferred-locale`。
8. `<html lang>`、canonical、`hreflang` 都从 `[locale]` 获取。
9. 新增语言只改 JSON 文件内容（每 leaf 补字段），不复制目录；同步改 `settings`/`labels`/`404 messages`。
10. 新增 namespace 同步在 `i18n-static.ts` 的 `MESSAGES` 注册，否则 `getI18nProps` 跳过。
11. 部署只上传 `out/` 目录。
